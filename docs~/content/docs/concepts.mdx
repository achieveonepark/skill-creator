## 실행 흐름

모든 스킬은 동일한 파이프라인을 따릅니다.

```text
Trigger → Targeting → Condition → Effect → Result
```

| 단계 | 담당 | 설명 |
| --- | --- | --- |
| Trigger | `SkillRunner` | 스킬 사용 요청. 쿨타임을 확인한다. |
| Targeting | `TargetResolver` | 타겟팅 정의로 실제 대상 목록을 계산한다. |
| Condition | `ConditionEvaluator` | 효과 실행 전 조건을 AND 로 검사한다. |
| Effect | `EffectExecutor` | 효과 목록을 순차 실행한다. |
| Result | — | 데미지/힐/버프 적용, 쿨타임 시작, 로그 기록. |

`SkillRunner.Use` 의 실제 순서는 다음과 같습니다.

```text
1. SkillDefinition 조회
2. 쿨타임 확인
3. 타겟 목록 계산
4. 조건 검사
5. (캐스팅 시간 대기 — UseRoutine 한정)
6. Effect 목록 순차 실행
7. 쿨타임 적용
8. 실행 로그 기록
```

## 작은 조각의 조합

스킬은 거대한 클래스가 아니라 데이터 조각의 조합입니다.

```text
Skill
 ├─ Targeting   (어디를 칠 것인가)
 ├─ Condition[] (실행해도 되는가)
 └─ Effect[]    (무엇을 할 것인가)
```

새로운 스킬은 **코드가 아니라 데이터**로 만들어집니다. 같은 `damage` / `add_buff` / `play_vfx` 조각을 다르게 조합하면 전혀 다른 스킬이 됩니다.

## 왜 ScriptableObject 를 최소화하나

`ScriptableObject` 는 Unity Editor 안에서는 편하지만, 콘텐츠 양산 도구로 보면 한계가 있습니다.

- 외부 툴(Spreadsheet/서버)과 연동이 불편함
- 런타임 패치가 까다로움
- 대량 데이터의 diff 확인이 불편함
- 기획자가 Unity Editor 를 반드시 열어야 함
- 운영 중 밸런스 수정 반영 흐름이 무거움

그래서 SkillForge 는 **JSON 을 1차 저장 형식**으로 사용하고, 다음 순서의 운용을 우선합니다.

```text
JSON → CSV → Google Spreadsheet Export → Remote Config → CDN → Addressables TextAsset
```

## 통합 경계

SkillForge 는 특정 전투 시스템을 강제하지 않습니다. 다음 인터페이스가 경계입니다.

| 인터페이스 | 역할 | 필수 |
| --- | --- | --- |
| `IBattleUnit` | 전투 유닛 추상화 | ✅ |
| `IUnitRegistry` | 광역 타겟팅용 유닛 조회 | circle/cone 사용 시 |
| `IVfxPlayer` / `ISfxPlayer` | 연출 재생 | 선택 |
| `ICombatLogger` | 실행 로그 | 선택 |

유닛이 가지는 `StatSheet`, `CooldownStore`, `BuffContainer` 는 SkillForge 가 제공하므로 직접 구현할 필요가 없습니다.

## MVP 범위

처음부터 노드 그래프나 완전한 수식 파서를 만들지 않습니다. MVP 는 **데이터를 만들고, 검증하고, 런타임에서 실행하는 것**까지입니다.

지원 타입은 [데이터 포맷](/data-format) 문서를 참고하세요.
